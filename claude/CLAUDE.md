# 指示

- 案を提示するときは、複数の案を推奨度と理由とともに提示してください。
- 仮定や推論を元に進める場合は、必ず、その方針で問題が無いか1件ずつ質問ツールを使って確認をとってください。
- こちらに確認が必要な場合は、1個ずつ質問ツールで質問してください。その時、質問が全部でいくつあるかもあわせて知らせてください
- チャットのやり取りを経てファイルの編集を行う場合、必ずこちらに確認を取ってください。
  - こちらは議論の継続や、編集前の計画立案を望んでいる場合があるからです。
  - 計画の承認は編集の承認ではありません。計画を承認しても、編集を開始する場合は確認してください。
- こちらが提示した手法に対して、目的を達成するためのより良い手法があれば提案してください。
  - 目的が不明な時は、確認してください。
- 常に日本語を使用すること

## コードコメント

- チャットでのやり取りを前提としたコメント（「案Aを選択」「議論の結果こうした」など）をコードに書かないこと。
  - 質問内容や他の選択肢はコードに残らないため、後から振り返れず無意味になるからです。
- 設計判断の経緯を記録する必要がある場合は、コードコメントではなく、コミットメッセージに「検討した選択肢・選ばなかった理由」を含めて記録すること。
- 自リポジトリのissueやPRの番号をコードコメントに書かないこと。経緯はgit blame→コミット→PRで辿れるため、コメントには番号ではなく理由そのものを書くこと。
  - 例外: 外部リポジトリのissueや仕様書など、git履歴から復元できない参照は、理由の文章とセットであれば書いてよい。

## Git

- あらたにブランチを作成するときは、指示されない限りupstreamを設定しないこと
- ファイルの編集作業をしているときは、意味のある単位で定期的にコミットを試みてください。

## 設計・実装

- 分割統治を意識すること
- SOLID原則を守ること
- プロパティベーステストが効果的なテストには、積極的にプロパティベーステストを採用する
  - `fast-check`を使う
- システム環境構築では、ローカルは使わず、Docker composeを前提にしてください。開発にはVSCodeのdevcontainerを使います。

## GitHub

- PullRequestを作成するときは必ずDraft PullRequestとして作成してください。

## Fable モデル使用時のルール

IMPORTANT: 以下のルールは、メインエージェント(このセッション自身)のモデルが
Claude Fable 系の場合にのみ適用する。それ以外のモデルの場合、このセクションは無視すること。

- メインエージェントは監督役に徹すること。コーディング・テスト・調査を自分で直接行わないこと。
  メインが直接行ってよいのは、タスクの分解・委任・結果のレビュー・統合・コミット・
  ユーザーとの対話のみ。
- 作業は必ず以下のサブエージェントに委任し、タスクの難易度に応じて使い分けること。
  - コーディング(通常): coder
  - コーディング(高難度: 設計を伴う変更、複雑なリファクタリング、難解なバグ): coder-hard
  - テストの作成・実行: tester
  - 調査(軽量: ファイル探索、単純な事実確認): researcher
  - 調査(高度: 仕様の深掘り、複数ソースの比較検討): researcher-deep
- サブエージェントに Fable 系モデルを使うことは禁止。上記エージェントの
  model 指定を inherit に変更しないこと。
- コミットはメインエージェントが行うこと。サブエージェントの完了報告をレビューした後、
  意味のある単位でコミットすること。サブエージェントにコミットさせないこと。
