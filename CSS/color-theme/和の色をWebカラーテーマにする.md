# 和の色をWebカラーテーマにする ―― Claude向け作業指示書

> この文書は、別の会話のClaudeに「日本の伝統色・自然物・文化的モチーフをWebサービスのカラーテーマに仕立てる」作業を、一定の品質で再現させるための指示書です。会話の冒頭にこの文書を貼り付け、続けて「この指示書に従って、〈テーマ名〉のカラーテーマを作って」と依頼してください。

---

## 0. この指示書の使い方（Claudeへ）

あなたはこれから、ユーザーが与える題材（例：平安の重ね色目「紅紅葉」、在来種ヤマアジサイ、特定の和歌の情景など）を、ブラウザで見られるビジュアルなカラーテーマ見本（HTML）に仕立てます。以下の手順と原則を順守してください。**勝手に全工程を走らせず、各段階で定められた確認を取ること。**

題材は和の色に限りません。同じ手順は任意のテーマ（季節、鉱物、料理、ブランドの世界観など）に応用できます。和の伝統色の場合は「典拠の正直さ」（§2）が特に重要になります。

---

## 1. 着手前のヒアリング（必須・1問ずつ）

作業を始める前に、**質問ツールを使って1問ずつ**確認する。一度に複数を聞かない。全体で何問あるかも伝える。最低限、次の3点は埋める（題材から自明なものは飛ばしてよい）。

1. **対象の範囲**：どの題材・どの色筋をテーマ化するか。候補が多い題材なら「広く調べてから選ぶ」か「指定の○個だけ」かを確認。
2. **テーマ数**：単一か複数か。複数なら何系統か（例：青系／紅系／移ろい系）。
3. **アウトプットの粒度**：ビジュアル見本のみか、調査メモも添えるか。コントラスト比表示の粒度（主要組み合わせのみ／網羅）。
ユーザーの設定（preferences）に「1問ずつ確認」「複数案は推奨度つきで提示」などがある場合は必ず従う。

---

## 2. 典拠を調べ、文化的に正直に扱う（和の題材で特に重要）

色名や色目の**由来・典拠を必ず確認する**。思い込みで「平安から伝わる」などと書かない。

- 伝統色・有職故実が絡むときは、信頼できる情報源（日本服飾史／綺陽装束研究所、長崎盛輝『かさねの色目』、博物館・大学資料、伝統色データベース等）を優先して調べる。
- **典拠が確認できない色目・色名は、その旨を明記する。** 例：このセッションでは「楓紅葉」「紫陽花の重ね色目」は古典原典に確認できず、近現代の色目集由来または現代の再構成である可能性が高いと判明した。こうした場合は、
  - テーマ名を実在を断言しない形にする（例：「あぢさゐの襲 ― 夏の青紫」「平安の夏のかさね色目に寄せて」）。
  - 画面内の「典拠についての注記」に、実在しない／後世由来である事実と、何を典拠に再構成したかを書く。
- 時代・流派で解釈が分かれる色は、その差異も書く（例：紅紅葉は平安原典ではグラデーション、中世武家系では真紅、と異なる）。
- 伝統色のHEX/RGBは**すべて「目安」**であり、史料の実色や植物の実色と厳密に一致しない旨を脚注・フッターに明記する。
- 「青」は平安では緑を指すなど、**語の歴史的意味のずれ**にも注意し、必要なら注記する。
調査結果は、テーマ設計の根拠として手元に整理しておく（ユーザーが調査メモも欲しい場合は別途まとめる）。

## 3. デザインの原則

### 3-1. 量産型の「AIっぽい和風」を避ける

「生成りクリーム地＋明朝体＋テラコッタ差し色」のような、AIが量産しがちなテンプレ的見た目に流れないこと。題材の**本質的な特性**をデザインの軸に据える。

- 例（重ね色目）：本質は「薄絹を重ね、下の色が透けて見える」こと → 署名要素を「袖口・裾からのぞく五衣の重なり」（層が少しずつ覗くスタック）にした。
- 例（ヤマアジサイ）：本質は「4枚の萼が集まる装飾花」「土壌で色が移ろう七変化」 → 署名要素を「少しずつ色違いの萼（floret）の集まり」にした。

### 3-2. 各テーマに「署名となるビジュアル」を1つ作る

単なる色チップの羅列で終わらせない。題材を象徴する図像をCSSで作り、ホバー等の軽いインタラクションを添える（`prefers-reduced-motion` を必ず尊重）。

### 3-3. 色を「役割（semantic role）」に割り当てる

彩りを並べるだけでなく、各色をUIの役割に対応させ、「Webに適用すると」の小さなUIモック（カード＋ボタン＋ピル）で見せる。役割は次の語彙で統一する：

```planetext
ground（地/背景）, surface（面）, primary（強調）, primary-btn（ボタン用・必要なら濃色）,
on-primary（ボタン上の文字）, secondary（副え）, secondary-ink（副え・文字用濃色）,
accent（差し色）, accent-ink（差し色・文字用濃色）, muted（中間）,
text（本文）, text-dim（淡い本文）
```

> **ライト版のコツ**：鮮やかな伝統色（山吹・紅・露草など）はライト地の上では文字に使うとコントラスト不足になりがち。「彩りチップ」には鮮やかな色をそのまま使い、**文字・見出し用には濃色の `-ink` 変数を別に用意**する。

## 4. 技術仕様（このセッションで確立した型）

成果物は**単一の自己完結HTMLファイル**（外部依存はGoogle Fontsのみ）。`/mnt/user-data/outputs/` に作成し、`present_files` で提示する。複数テーマは1ファイルに縦／横並びで収める。別題材は別ファイルにする。

### 4-1. ダーク/ライト切替

- モード変数（`--page-bg` 等のページ全体の色を含む）は **`<html>` 要素**に `html[data-mode="dark"]` / `html[data-mode="light"]` で定義する。**`body` に定義しない**（後述の背景バグの原因になる）。
- 初期値として `<html lang="ja" data-mode="dark">` のように属性を付ける。
- 切替JSは `document.documentElement.dataset.mode` を書き換える。`body` ではない。
- テーマ固有のトークンは `.theme.xxx { --color-...: ... }`、ライト版は `html[data-mode="light"] .theme.xxx { ... }` で上書き。

### 4-2. 背景が切り替わらないバグの回避（重要・つまずきポイント）

プレビュー環境（iframe）では `<html>` の背景が描画されず、アプリの色が透けて「背景が変わらない」ように見えることがある。これを確実に防ぐため、**ビューポート全体を覆う固定背景レイヤー**を必ず入れる：

```css
html{min-height:100%;background:var(--page-bg);transition:background .5s ease}
/* 描画環境に依存せず確実に切り替えるための固定レイヤー */
body::before{content:"";position:fixed;inset:0;z-index:-1;
  background:var(--page-bg);transition:background .5s ease}
body{background:var(--page-bg);min-height:100vh;color:var(--page-ink)}
```

`color-mix()` を使う箇所（半透明トグルバー等）は**必ず単色フォールバックを前置**する：

```css
background:var(--panel);                                  /* フォールバック */
background:color-mix(in srgb,var(--page-bg) 88%,transparent);
```

> 補足：このバグはユーザーから2回指摘された。最初は変数を `body` から `html` に移して半分直り、最終的に `body::before` の固定レイヤーで確実に解決した。**最初からこの型を使えば再発しない。**

### 4-3. WCAGコントラスト比の表示と検証

- 主要な組み合わせ（本文×地／本文×面／主ボタン／副え見出し×地／差し色×地）について、コントラスト比とAA/AAA判定を画面に表示する。
- 判定基準：**本文 4.5:1（AAAは7.0）／大きい文字・UI部品 3.0:1（AAAは4.5）**。
- 比率は**JavaScriptで、表示中モードの実際のCSS変数値から算出**する（モード切替時に再計算）。固定値を書き込まない。これにより表示が常に実態と一致する。
- **作る前に**、決めた色で全組み合わせ・全モードがAA以上になるかをコード（bash/python等）で検証してから着手する。NGがあれば色を調整（例：白文字ボタンが4.5未満なら背景色を濃くする）。
参考：相対輝度とコントラスト比の計算（WCAG式）

```js
function lum(hex){
  const [r,g,b]=hex.replace('#','').match(/../g).map(h=>{
    let v=parseInt(h,16)/255; return v<=0.03928?v/12.92:Math.pow((v+0.055)/1.055,2.4);
  });
  return 0.2126*r+0.7152*g+0.0722*b;
}
function ratio(a,b){const l1=lum(a),l2=lum(b),hi=Math.max(l1,l2),lo=Math.min(l1,l2);
  return (hi+0.05)/(lo+0.05);}
```

### 4-4. CSS変数（Design Tokens）の書き出し

- ページ下部に、開発へ貼り付け可能な形でトークンを出力し、「コピー」ボタンを付ける。
- 形式は `[data-theme="..."]{ --color-...: ... }` と `[data-theme="..."][data-mode="light"]{ ... }`。`<html>` でも `<body>` でも任意要素に属性を付けて使える汎用形にする。
- ユーザーがDocker/devcontainer前提の開発をしている場合がある旨を添える。

### 4-5. その他の実装ルール

- フォント：日本語表示なら明朝（例 Shippori Mincho B1）＋ゴシック（例 Zen Kaku Gothic New）の2種程度。
- `localStorage`/`sessionStorage` は使わない（プレビュー環境で動かない）。状態はDOM属性やJS変数で持つ。
- レスポンシブ対応（狭幅でテーマを縦積みに）。
- 打ち間違い（HEXに空白が入る等）に注意し、作成後に `grep` 等で検査する。

## 5. 仕上げと確認

- 完成したら `present_files` で提示し、何をしたか要点を簡潔に説明する。
- ユーザー設定に「編集前に確認」「複数案は推奨度つき」等があれば従う。**計画の承認と編集の承認は別**。編集を始める前に確認を取る。
- 改善余地は**推奨度（高/中/低）と理由つきで複数案**として提示し、ユーザーに選んでもらう。
- 常に日本語で応答する（ユーザーが日本語の場合）。
